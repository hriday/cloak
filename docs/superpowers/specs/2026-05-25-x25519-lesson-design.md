# X25519 Lesson — Design

**Date:** 2026-05-25
**Project:** `cloak`
**Working directory:** /Users/hriday/code/enc_algo

## Goal

Teach X25519 as an 8-step lesson that builds key exchange from the ground up: classical Diffie-Hellman first (modular exponentiation with small primes), then ECDH conceptually (same pattern on a different group), then X25519 specifically (the Curve25519 choices, the clamping ritual, the 32-byte u-coordinate world). The lesson ends with a live Web Crypto exchange between two in-page keypairs and a Python codegen using `cryptography.hazmat.primitives.asymmetric.x25519`.

This is the site's first key-exchange lesson. Every prior asymmetric algorithm (RSA, hybrid) used the public key to *encrypt*. X25519 is the canonical example of a key-agreement primitive: nothing is ever encrypted with the public key — the two sides each compute the same secret from their own private key and the other's public key. That conceptual shift is the load-bearing pedagogy here.

## Curriculum position

1. RSA ✅
2. Hybrid Encryption ✅
3. AES ✅
4. Triple DES (designed)
5. Blowfish (designed)
6. Twofish (designed)
7. HSM (designed)
8. **X25519** (this design)
9. Ed25519 (future, pk=11 — signatures on the same curve family)
10. Kyber / ML-KEM (future PQ-KEM lesson)
11. Caesar / Vigenère (future)

Pedagogically: comes after the symmetric block-cipher arc so learners already understand "shared secret + symmetric cipher" as the goal X25519 enables. The natural follow-on is Ed25519 (signatures on Curve25519's twisted Edwards form), then a post-quantum KEM lesson framed as "what replaces X25519 when Shor's algorithm shows up."

## Non-goals

- Full elliptic-curve arithmetic. The lesson gives geometric intuition for point addition + scalar multiplication; no point-addition formulas, no chord-and-tangent diagrams worked algebraically.
- The Montgomery ladder. Mentioned as the algorithm used for `scalar * point`; not walked.
- The full RFC 7748 decode/encode routines (little-endian byte order, the u-coordinate recovery dance). Mentioned as "RFC 7748 specifies exact byte encodings"; not executed by hand.
- Ed25519 (separate future lesson, pk=11). Mentioned only as a forward-link.
- X448 — mentioned as the higher-security cousin (Curve448, ~224-bit security) but not taught.
- HKDF after the raw shared secret. Real-world X25519 outputs always go through a KDF before keying a symmetric cipher; deferred to a future "HKDF + KDFs" lesson. Called out in the recap, not taught here.
- Quantum security: noted that Shor's breaks both DH and ECDH; motivates the future Kyber lesson. Not taught.
- Small-subgroup attacks beyond a one-sentence mention of why clamping exists.

## User experience

The lesson lives at `/algorithms/x25519/learn/key-exchange-on-a-curve/`. X25519 appears as a new card on the landing page. `family: asymmetric` (reusing the existing family — even though it's key exchange, the keys are asymmetric; adding a `kex` family would touch the model for one algorithm).

### Step sequence

| # | Slug | Kind | Behavior |
|---|---|---|---|
| 1 | `intro` | `info` | The problem: two strangers on a public network want a shared secret. Eve sees every byte. How do they agree without ever transmitting the secret? Forward-references: this primitive is *the* enabler of HTTPS (TLS 1.3 handshake), Signal, WireGuard, SSH. Without it, none of those exist. |
| 2 | `classical-dh` | `info` | Diffie-Hellman 1976. Public params: prime `p`, generator `g`. Alice picks secret `a`, sends `A = g^a mod p`. Bob picks secret `b`, sends `B = g^b mod p`. Both compute `s = B^a = A^b = g^(ab) mod p`. Eve sees `A` and `B` but can't compute `g^(ab)` without solving discrete log. Worked example with small primes: p=23, g=5, a=6, b=15 → A=8, B=19, s=2. |
| 3 | `do-a-dh` | `input-numeric` | Interactive. Page fixes p=23, g=5, a=6 (computes A=8), b=15 (computes B=19). User computes the shared secret `s = B^a mod p = 19^6 mod 23`. Validator accepts 2. Hint walks the modular exponentiation. |
| 4 | `why-curves` | `info` | Why move from mod-p DH to elliptic curves. Two reasons: smaller keys for equivalent security (256-bit ECC ≈ 3072-bit DH), and faster scalar ops. Intro to elliptic-curve groups: points `(x, y)` on a curve, with point-addition and scalar-multiplication defined geometrically. Emphasizes the structural point — "same DH pattern, different group" — without working any formulas. |
| 5 | `curve25519-spec` | `info` | The specific curve: `y² = x³ + 486662x² + x mod (2²⁵⁵ − 19)`, Montgomery form, designed by Daniel Bernstein in 2005 (eprint 2006/006, published 2006). The base point's u-coordinate is `9`. Private keys are 32 random bytes, *clamped* (bits 0,1,2 of byte 0 cleared; bit 6 of byte 31 set; bit 7 of byte 31 cleared) — this prevents small-subgroup attacks and forces a fixed-bit-length scalar so the Montgomery ladder runs in constant time. Public keys are 32 bytes (u-coordinate only). Shared secret = u-coordinate of `Alice_priv · Bob_pub` (= `Bob_priv · Alice_pub`). |
| 6 | `clamping` | `input-numeric` | Interactive. Page shows a fixed-for-the-lesson byte-0 value (e.g. `0xB7`) and a byte-31 value (e.g. `0x4F`). User computes the clamped result for byte 0 (`AND 0xF8`) and submits it in hex. Validator confirms `0xB7 & 0xF8 = 0xB0`. Hint walks the bit pattern. Step also displays the byte-31 clamping rule (`AND 0x7F, OR 0x40`) for context but only checks byte 0. |
| 7 | `exchange-keys` | `input-text` | Interactive. Page calls Web Crypto: `crypto.subtle.generateKey({name:"X25519"}, true, ["deriveBits"])` twice (Alice + Bob), exports both public keys as raw bytes, derives the shared secret from both directions, and renders all three blobs as hex. The two derived secrets are shown side-by-side; the page asserts they match and highlights the agreement. User types the literal string `match` (or similar) to advance. Validator confirms agreement was observed and writes the actual blobs into state. | 8 | `done` | `info` | Recap. Forward-links: Ed25519 (signatures on the twisted-Edwards form of the same curve), HKDF (what real systems put the shared secret through before keying a cipher), Kyber / ML-KEM (the post-quantum replacement when Shor's algorithm lands). Full Python script using `cryptography.hazmat.primitives.asymmetric.x25519`. |

### Step 3 in detail — `do-a-dh`

Prompt:

> Alice and Bob have agreed on public parameters `p = 23`, `g = 5`. Alice picked her secret `a = 6` and sent Bob `A = g^a mod p = 5^6 mod 23 = 8`. Bob picked his secret `b = 15` and sent Alice `B = g^b mod p = 5^15 mod 23 = 19`.
>
> You are Alice. Compute the shared secret `s = B^a mod p = 19^6 mod 23`.

Validator: parses an integer, equals `2`. Hint: `"19² mod 23 = 361 mod 23 = 16. 19⁴ = 16² mod 23 = 256 mod 23 = 3. 19⁶ = 19⁴ · 19² mod 23 = 3 · 16 mod 23 = 48 mod 23 = 2."`

### Step 6 in detail — `clamping`

Prompt:

> A fresh X25519 private key is 32 random bytes, but before use the spec *clamps* three bits to defend against small-subgroup attacks and pin the high bit so the Montgomery ladder runs in constant time. The clamping rules (RFC 7748):
>
> - Byte 0: clear bits 0, 1, 2 → `byte0 = byte0 AND 0xF8`
> - Byte 31: clear bit 7 → `byte31 = byte31 AND 0x7F`
> - Byte 31: set bit 6 → `byte31 = byte31 OR 0x40`
>
> Your random byte 0 is `0xB7` (binary `10110111`). Apply the byte-0 clamping rule and enter the result in hex.

Validator: parses hex/decimal, equals `0xB0`. Hint: `"0xB7 = 10110111. Clear the bottom three bits (mask with 0xF8 = 11111000): 10110000 = 0xB0."`

The page also renders the byte-31 result (`0x4F AND 0x7F OR 0x40 = 0x4F`, which happens to be unchanged for that value) so the learner sees the full ritual without being asked to compute both.

### Step 7 in detail — `exchange-keys`

Behavior:

1. On step enter, JS calls Web Crypto: generate Alice's keypair, generate Bob's keypair (both `{name: "X25519"}`).
2. Export both public keys as raw bytes (`exportKey("raw", ...)`).
3. Derive `s_AB = Alice_priv · Bob_pub`, `s_BA = Bob_priv · Alice_pub` (`deriveBits` with `{name: "X25519", public: peerPub}`).
4. Render: Alice's pubkey (hex), Bob's pubkey (hex), `s_AB` (hex), `s_BA` (hex). Compare `s_AB === s_BA` and render a green "agreement" panel or red mismatch.
5. User confirms by typing `match` (case-insensitive) into a single text input.

Validator `exchange_keys`:
- Confirms the page-rendered secrets matched (a boolean stashed on the Alpine component when the derivation ran).
- Confirms user input equals `match`.
- Writes `{x25_alice_pub, x25_bob_pub, x25_shared_secret}` (the hex strings) into state.

Fallback when Web Crypto X25519 is unavailable:
- The static module ships a canonical demo set: `{alice_pub, bob_pub, shared_secret}` precomputed from a fixed seed. The page renders these instead with a banner: `"Your browser doesn't support X25519 in Web Crypto. Showing canonical RFC 7748 test vectors — the math is identical."`
- The validator still accepts `match` since the canonical vectors do agree by construction.
- Feature-detection: `try { await crypto.subtle.generateKey({name:"X25519"},true,["deriveBits"]); } catch { /* fallback */ }`.

## Data flow

```
Step 1, 2 (info)         no input. state empty.

Step 3 (do-a-dh)         input: integer
  ↓ compute_dh validator
state += { x25_classical_shared: 2 }

Step 4, 5 (info)         no input. Step 5 may interpolate x25_classical_shared in
                         a "now scale this up from p=23 to 2²⁵⁵-19" lead-in.

Step 6 (clamping)        input: hex/decimal byte
  ↓ clamp_byte validator
state += { x25_clamped: "0xB0" }

Step 7 (exchange-keys)   side effect: Web Crypto generate + derive runs on step enter
                         input: text ("match")
  ↓ exchange_keys validator
state += { x25_alice_pub, x25_bob_pub, x25_shared_secret }

Step 8 (done)            info. Codegen uses everything in state for the final script.
```

## Codegen

Per-step lines build up to a single runnable script. Final `full_script` (step 8) target:

```python
from cryptography.hazmat.primitives.asymmetric import x25519
from cryptography.hazmat.primitives import serialization

alice_priv = x25519.X25519PrivateKey.generate()
bob_priv   = x25519.X25519PrivateKey.generate()

alice_pub_bytes = alice_priv.public_key().public_bytes(
    encoding=serialization.Encoding.Raw,
    format=serialization.PublicFormat.Raw,
)
bob_pub_bytes = bob_priv.public_key().public_bytes(
    encoding=serialization.Encoding.Raw,
    format=serialization.PublicFormat.Raw,
)

s_ab = alice_priv.exchange(bob_priv.public_key())
s_ba = bob_priv.exchange(alice_priv.public_key())

print("Alice pub :", alice_pub_bytes.hex())
print("Bob pub   :", bob_pub_bytes.hex())
print("s_AB      :", s_ab.hex())
print("s_BA      :", s_ba.hex())
assert s_ab == s_ba, "shared secrets must agree"
```

The codegen module emits this script with the *live* Alice/Bob/shared hex values from `state` as comments next to the relevant lines, so the script the learner downloads and the page they saw cross-reference cleanly.

## Error handling

### Step 3 — `compute_dh`
- Not parseable → `"Enter a whole number."`
- Wrong value → `"Compute 19^6 mod 23. Hint: 19² mod 23 = 16, 19⁴ mod 23 = 3, 19⁶ = 19⁴ · 19² mod 23 = 48 mod 23 = 2."`

### Step 6 — `clamp_byte`
- Not parseable → `"Enter a byte in hex (0xNN) or decimal (0–255)."`
- Out of [0, 255] → `"A byte is 0–255."`
- Wrong value → `"Clear the bottom three bits of 0xB7. 0xB7 AND 0xF8 = 0xB0."`

### Step 7 — `exchange_keys`
- Page derivation hasn't run yet → `"Wait for the exchange to finish — both secrets should appear above."`
- Secrets didn't match (shouldn't happen, but defensively) → `"The browser reported a mismatch. Refresh the step to retry."`
- User typed something other than `match` → `"Type 'match' to confirm both sides derived the same secret."`

## Open questions / deferred

- **Web Crypto X25519 availability.** Strategy is *feature-detected with a canonical fallback*. Chrome 110+ (Feb 2023), Firefox 125+ (Apr 2024), Safari 17+ (Sep 2023) all ship it; older browsers get a frozen demo. The fallback uses RFC 7748 §6.1 test vectors so the bytes shown are pedagogically canonical, not made-up.
- **HKDF after the raw shared secret.** Real systems never feed raw X25519 output to a symmetric cipher; they HKDF-Expand it first. Mentioned in step 8's recap, deferred to a future "HKDF + KDFs" lesson.
- **Encoding direction.** RFC 7748 specifies little-endian byte order for both scalars and u-coordinates. Mentioned in step 5 prose; not exercised. A future "RFC archaeology" step could walk the encode/decode if learners ask for it.
- **Why this curve at all.** Curve25519's design choices (low-weight prime, specific A coefficient, twist-secure base point) are a rabbit hole; the lesson stops at "Bernstein 2005, chosen for speed + safety, widely audited" and leaves the cryptographic-engineering deep-dive for a future spec.
- **Quantum.** Shor's breaks ECDH the same way it breaks RSA. The recap calls this out and forward-links to a future Kyber / ML-KEM lesson. Not taught here.
